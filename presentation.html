The Problem

The Reality of Writing OpenAPI by Hand

# openapi.yaml – a "simple" endpoint
paths:
  /pets/{id}:
    get:
      summary: Get a pet by ID
      parameters:
        - name: id
          in: path
          required: true
          schema:
            type: string
      responses:
        '200':
          description: A pet object
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/Pet'
        '404':
          description: Not found

components:
  schemas:
    Pet:
      type: object
      required: [id, name]
      properties:
        id:
          type: string
        name:
          type: string
        age:
          type: integer

😩 Verbosity

20+ lines to describe one GET endpoint. Real APIs have hundreds.

🔁 Duplication

Every shared model must be repeated or referenced with fragile $ref strings.

🧩 No abstraction

No generics, no inheritance, no templates. Copy-paste is the "pattern".

🤯 Hard to review

Large YAML files are unreadable in PRs and nearly impossible to diff.

The Problem

More Pain Points at Scale

Consistency

10 teams → 10 different naming conventions, error shapes, pagination styles. No enforced standard.

Multi-protocol

OpenAPI only covers REST. gRPC, AsyncAPI, JSON Schema need completely separate files.

Tooling drift

Generated clients get out of sync. Different validators disagree on the same spec. Breaking changes slip through.

No type safety

YAML/JSON is text. Typos in $ref, wrong types, missing fields — all caught only at runtime.

Version sprawl

OpenAPI 2 vs 3.0 vs 3.1 — incompatibilities everywhere. Swagger UI, ReDoc, Redocly all behave differently.

Scaling

Microsoft's Azure REST APIs: thousands of endpoints. Managing raw OpenAPI at that scale is untenable.

💡 TypeSpec was born from exactly this pain — at Microsoft, managing the Azure API surface.

🔷

What is TypeSpec?

A language for describing APIs — not just documenting them

What is TypeSpec

TypeSpec at a Glance

TypeSpec is an open-source, Microsoft-built language for describing cloud service APIs and shapes — then generating everything from that single source.

  • 🧠Strongly-typed, compiler-verified API definition language
  • ⚙️Compiles to OpenAPI 3, JSON Schema, Protobuf, and more
  • 🔌Extensible via libraries and custom emitters
  • 🏭Used internally at Microsoft to define the entire Azure API surface
  • 🌐Open source at github.com/microsoft/typespec
typespec.io

The Core Idea

Write your API once in TypeSpec.
Let the compiler produce OpenAPI, JSON Schema, docs, and client SDKs.

Inspired by TypeScript

Familiar syntax. If you know TypeScript, TypeSpec feels immediately natural.

Not a runtime framework

TypeSpec is a design-time tool. It doesn't run your API — it defines its contract.

Approach

The "Code-First" Approach with TypeSpec

Traditional approaches to API design:

Design-First (old style)

Write YAML/JSON OpenAPI spec by hand → implement server → hope they stay in sync.

Code-First (old style)

Write server code → annotate with attributes → generate OpenAPI. Spec is a by-product, often imperfect.

TypeSpec Code-First

Write TypeSpec — a real programming language. Compile → get perfect, consistent OpenAPI (and more). The spec is the source of truth.

Why Code-First with TypeSpec wins

  • Refactor-safe: rename a model, everywhere updates
  • IDE support: autocomplete, go-to-definition, error squiggles
  • Reuse via imports, templates, libraries
  • Version control friendly: small, readable diffs
  • Enforces API style through shared libraries (like linting)
  • One input → multiple outputs (OpenAPI, Protobuf, docs…)

Core Syntax Features

TypeSpec vs OpenAPI — side by side

Core Syntax – Models

Models → OpenAPI Schemas

TypeSpec

model Pet {
  id:   string;
  name: string;
  age?: uint8 = 0;
  kind: "cat" | "dog" | "bird";
}

model Dog extends Pet {
  breed: string;
}

Equivalent OpenAPI 3

components:
  schemas:
    Pet:
      type: object
      required: [id, name, kind]
      properties:
        id:   { type: string }
        name: { type: string }
        age:
          type: integer
          format: uint8
          default: 0
        kind:
          type: string
          enum: [cat, dog, bird]
    Dog:
      allOf:
        - $ref: '#/components/schemas/Pet'
        - type: object
          required: [breed]
          properties:
            breed: { type: string }

TypeSpec: 8 lines  |  OpenAPI YAML: 22 lines — and TypeSpec is statically checked.

Core Syntax – Operations

Operations → OpenAPI Paths

TypeSpec

import "@typespec/http";
using TypeSpec.Http;

@route("/pets")
namespace Pets {

  @get
  op listPets(): Pet[];

  @post
  op createPet(@body pet: Pet): Pet;

  @get @route("/{id}")
  op getPet(@path id: string):
    Pet | NotFoundError;

  @delete @route("/{id}")
  op deletePet(@path id: string): void;
}

Generated OpenAPI (abbreviated)

paths:
  /pets:
    get:
      operationId: listPets
      responses:
        '200':
          content:
            application/json:
              schema:
                type: array
                items:
                  $ref: '#/components/schemas/Pet'
    post:
      operationId: createPet
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/Pet'
      responses:
        '200': ...
  /pets/{id}:
    get:  ...
    delete: ...
Core Syntax – Decorators

Decorators — Metadata as First-Class Citizens

Decorators attach HTTP, validation, and documentation metadata directly to types.

@doc("A registered pet")
model Pet {
  @doc("Unique identifier")
  @key
  id: string;

  @minLength(1) @maxLength(50)
  name: string;

  @minValue(0) @maxValue(30)
  age?: uint8;
}

Common built-in decorators

DecoratorEffect
@docdescription in generated spec
@routeURL path segment
@get @post @put @delete @patchHTTP verb
@path @query @header @body @cookieparameter location
@minLength @maxLengthstring constraints
@minValue @maxValuenumeric constraints
@formate.g. "uuid", "date-time"
@keymarks the resource identifier
Core Syntax – Interfaces

Interfaces & Namespaces — Organise at Scale

Grouping operations with interface

@route("/pets")
interface PetsService {
  @get
  op list(): Pet[];

  @post
  op create(@body b: CreatePetRequest):
    Pet;
}

// Separate namespace for admin routes
@route("/admin")
namespace Admin {
  @get @route("/pets")
  op listAll(): Pet[];
}

namespace

Logical grouping. Translates to a path prefix. Nestable. Mirrors how large API surfaces are split into domains.

interface

Groups a set of operations that share behaviour or a resource. Can be extended or referenced from templates — just like an interface in OO languages.

Combining both

Real APIs: namespace per domain (Users, Orders, Products), interface per resource within the domain.

Core Syntax – Templates

Templates — Generics for APIs

Define once, use for every resource

// Generic paged response
model Page<T> {
  items: T[];
  nextLink?: string;
  totalCount: int32;
}

// Generic CRUD interface
interface CrudOps<TResource, TCreate> {
  @get    op list(): Page<TResource>;
  @post   op create(@body b: TCreate): TResource;
  @get   @route("/{id}")
          op read(@path id: string): TResource;
}

// Instantiate for concrete resources
@route("/pets")
interface Pets extends CrudOps<Pet, CreatePet> {}

Zero duplication

10 resources → 10 one-liner interfaces. All pagination, error handling, and CRUD shapes defined once.

Compiler-checked

If you forget a field in a model passed to the template, you get a compile error — not a broken spec.

OpenAPI has no equivalent

Raw OpenAPI forces copy-paste for every resource. TypeSpec's templates are a transformative advantage.

Core Syntax – Unions & Errors

Union Types & Structured Error Responses

// Standard error envelope
model ApiError {
  code:    string;
  message: string;
}

@error
model NotFound    is ApiError;
@error
model Unauthorized is ApiError;
@error
model Conflict     is ApiError;

// Return type is a union
op getPet(@path id: string):
  Pet | NotFound | Unauthorized;

Union types

Operations can return multiple response types. TypeSpec maps each to the appropriate HTTP status code automatically when combined with @statusCode.

@error decorator

Marks a model as an error response. The emitter knows to map it to 4xx/5xx status codes in OpenAPI.

Consistent errors, always

Define ApiError once in a shared library. Every team, every service uses the same error shape — enforced by the compiler.

Core Syntax – Scalars & Enums

Scalars, Enums & Type Aliases

Custom scalars (branded types)

// Extend built-in scalars
@format("uuid")
scalar Uuid extends string;

@minLength(1) @maxLength(255)
scalar NonEmptyString extends string;

// Enums
enum PetKind {
  Cat: "cat",
  Dog: "dog",
  Bird: "bird",
}

model Pet {
  id:   Uuid;
  name: NonEmptyString;
  kind: PetKind;
}

Scalars

Create domain-specific types that carry validation rules. Use Uuid everywhere → impossible to accidentally pass a bare string.

Enums

Named, strongly-typed enumerations. Compile to OpenAPI enum arrays. IDE gives you autocomplete on enum values.

vs OpenAPI

OpenAPI has no custom scalars. You'd have to copy format: "uuid" and minLength: 1 onto every single field. Forever.

The TypeSpec Ecosystem

Libraries, emitters, tooling, and integrations

Ecosystem

Official Libraries

@typespec/http

HTTP routing decorators: @route, @get, @post, @body, @path, @query, @header

@typespec/rest

REST resource patterns: Resource, CollectionWithNextLink. Builds on @typespec/http.

@typespec/openapi3

Emitter that produces OpenAPI 3.0/3.1 YAML or JSON from your TypeSpec.

@typespec/versioning

Declare API versions. Mark properties as added/removed in specific versions. Generates per-version OpenAPI.

@typespec/json-schema

Emit pure JSON Schema from TypeSpec models — useful for message validation, config schemas.

@typespec/protobuf

Emit .proto files for gRPC services from the same TypeSpec source.

@azure-tools/typespec-azure-core

Azure-specific patterns. Used internally at Microsoft to define the entire Azure REST API surface.

@typespec/compiler

The core compiler. Provides the LSP server, CLI (tsp), formatter, and extension APIs.

Community

Growing ecosystem of community emitters, linters, and style libraries published on npm.

Ecosystem – Tooling

Developer Tooling

VS Code Extension

Full Language Server Protocol support: syntax highlighting, autocomplete, go-to-definition, inline errors, hover docs, formatter.

Visual Studio Extension

Same LSP features for the .NET ecosystem.

TypeSpec Playground

Online editor at typespec.io/playground — write TypeSpec, see OpenAPI output instantly. No install needed. Great for demos.

CLI — tsp

# Scaffold a new project
tsp init

# Compile & emit
tsp compile .

# Watch mode
tsp compile . --watch

# Format all .tsp files
tsp format **/*.tsp

tspconfig.yaml

Controls which emitters run, output paths, compiler options — per-project, version-controlled.

Ecosystem – Versioning

API Versioning — Built-In

import "@typespec/versioning";
using TypeSpec.Versioning;

@versioned(Versions)
namespace PetStore {

  enum Versions { v1, v2 }

  model Pet {
    id:   string;
    name: string;

    // Added in v2 — invisible in v1 spec
    @added(Versions.v2)
    microchipId?: string;
  }

  // Whole operation added in v2
  @added(Versions.v2)
  @get @route("/pets/microchip/{id}")
  op findByMicrochip(@path id: string): Pet;
}

One source — multiple versions

The emitter generates a separate OpenAPI file per version. The diff between versions is tracked in the TypeSpec source.

Available annotations

  • @added(version) — when a field/op appeared
  • @removed(version) — when deprecated/removed
  • @renamedFrom(version, oldName)
  • @madeOptional(version)

vs OpenAPI

Raw OpenAPI: maintain two completely separate, manually synced YAML files. TypeSpec: single file, compiler does the rest.

Summary

Key Takeaways

The problem

Raw OpenAPI YAML is verbose, fragile, doesn't scale, and forces copy-paste patterns across teams.

What TypeSpec is

A strongly-typed, compiler-checked language that generates OpenAPI and more from a clean, concise source.

Code-first done right

The TypeSpec file is the source of truth — not the generated YAML. Refactoring, versioning, and consistency become trivial.

Killer features

Generics/templates, custom scalars, decorators, built-in versioning, multi-protocol output.

Ecosystem

Official emitters for OpenAPI, JSON Schema, Protobuf. VS Code extension with full IDE support. Growing npm ecosystem.

Hands-On — Step 2

Define Your First Model

Replace main.tsp with:

import "@typespec/http";
using Http;

@service(#{ title: "Pet Store API" })
namespace PetStore;

model Pet {
  @doc("Unique pet identifier")
  id: string;

  @minLength(1)
  name: string;

  kind: "cat" | "dog" | "bird";
  age?: uint8;
}

model CreatePetRequest {
  @minLength(1)
  name: string;

  kind: "cat" | "dog" | "bird";
  age?: uint8;
}

  • Autocomplete on types (uint8 etc.)
  • Hover over @minLength to see docs
  • Introduce a type error — see red squiggle

Compile & inspect

Show the components/schemas section.

Use tsp compile . --watch for live output while editing.

Hands-On — Step 3

Add CRUD Operations

Add below the models in main.tsp:

@route("/pets")
interface Pets {

  @get
  op listPets(): Pet[];

  @post
  op createPet(
    @body body: CreatePetRequest
  ): Pet;

  @get @route("/{id}")
  op getPet(@path id: string): Pet;

  @put @route("/{id}")
  op updatePet(
    @path id: string,
    @body body: CreatePetRequest
  ): Pet;

  @delete @route("/{id}")
  op deletePet(@path id: string): void;
}

What to show

  • Recompile and show generated paths in OpenAPI
  • Count lines: ~30 TypeSpec vs ~120 OpenAPI YAML
  • Open typespec.io/playground — paste the code, see output live

Discussion point

What if you need to add @query filter params to listPets? Show how simple it is vs editing raw YAML.

Hands-On — Step 4

Add Error Handling

Add error models & update operations

@error
model NotFoundError {
  @statusCode code: 404;
  message: string;
}

@error
model ValidationError {
  @statusCode code: 400;
  message: string;
  fields: string[];
}

// Update getPet to return union
@get @route("/{id}")
op getPet(@path id: string):
  Pet | NotFoundError;

// Update createPet
@post
op createPet(@body b: CreatePetRequest):
  Pet | ValidationError;

Show in generated OpenAPI

The union types map to separate responses entries with the correct HTTP status codes — automatically.

Consistency win

Define NotFoundError once in a shared file. Import it everywhere. All 404s look the same across all endpoints.

Next step idea

Extract models into a models.tsp file and use import "./models.tsp" to show modular project structure.

Hands-On — Step 5

Templates & Reuse (Bonus)

Introduce a reusable paged response

// shared/pagination.tsp
model Page<T> {
  items: T[];
  totalCount: int32;
  nextLink?: string;
}

// Update listPets in main.tsp
import "./shared/pagination.tsp";

@get
op listPets(
  @query skip?: int32,
  @query top?:  int32
): Page<Pet>;

The pay-off

Add a second resource (e.g. Owner). The same Page<T> works instantly — zero duplication, compiler-verified.
